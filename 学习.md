#### spec各个阶段

%prep 读取位于 %sourcedir 目录的源代码和 patch 之后，解压源代码至 %builddir 的子目录并应用所有 patch。

%build 编译位于 %_builddir 构建目录下的文件。通过执行类似 ./configure && make 的命令实现。

%install 读取位于 %builddir 构建目录下的文件并将其安装至 %buildrootdir 目录。这些文件就是用户安装 RPM 后，最终得到的文件。

%check 检查软件是否正常运行。通过执行类似 make test 的命令实现。

bin  读取位于 %buildrootdir 最终安装目录下的文件，以便最终在 %rpmdir 目录下创建 RPM 包。在该目录下，不同架构的 RPM 包会分别保存至不同子目录， noarch 目录保存适用于所有架构的 RPM 包。这些 RPM 文件就是用户最终安装的 RPM 包。

src  创建源码 RPM 包（简称 SRPM，以.src.rpm 作为后缀名），并保存至 %_srcrpmdir 目录。SRPM 包通常用于审核和升级软件包。

#### 添加批量用户 

```shell
1. 创建目录 

 sudo mkdir /home/user1

2. 创建文本信息

sudo vi /home/user1/users.txt

3. 创建附加组

sudo groupadd group1

4. 编写脚本 用于创建用户

\#!/bin/bash

while IFS=: read -r username password uid groups homedir shell; do

  echo "Creating user: $username"

  sudo useradd -m -p "$password" -u "$uid" -g "$groups" -d "$homedir" -s "$shell" "$username"

  sudo chpasswd <<<"$username:$password"

  sudo passwd -e "$username"

  sudo chown "$username":"$groups" "$homedir"

done < users.txt

5.执行脚本

./create_users.sh /path/to/users.txt
```

另外，还有几个特殊字符用来处理参数：

| 参数处理 | 说明                                                         |
| :------- | ------------------------------------------------------------ |
| $#       | 传递到脚本的参数个数                                         |
| $*       | 以一个单字符串显示所有向脚本传递的参数。 如"$*"用「"」括起来的情况、以"$1 $2 … $n"的形式输出所有参数。 |
| $$       | 脚本运行的当前进程ID号                                       |
| $!       | 后台运行的最后一个进程的ID号                                 |
| $@       | 与$*相同，但是使用时加引号，并在引号中返回每个参数。 如"$@"用「"」括起来的情况、以"$1" "$2" … "$n" 的形式输出所有参数。 |

$* 与 $@ 区别：

- 相同点：都是引用所有参数。

- 不同点：只有在双引号中体现出来。假设在脚本运行时写了三个参数 1、2、3，则 " * " 等价于 "1 2 3"（传递了一个参数），而 "@" 等价于 "1" "2" "3"（传递了三个参数）。

#### Shell 输入/输出重定向

大多数 UNIX 系统命令从你的终端接受输入并将所产生的输出发送回到您的终端。一个命令通常从一个叫标准输入的地方读取输入，默认情况下，这恰好是你的终端。同样，一个命令通常将其输出写入到标准输出，默认情况下，这也是你的终端。

重定向命令列表如下：

| 命令            | 说明                                               |
| :-------------- | :------------------------------------------------- |
| command > file  | 将输出重定向到 file。                              |
| command < file  | 将输入重定向到 file。                              |
| command >> file | 将输出以追加的方式重定向到 file。                  |
| n > file        | 将文件描述符为 n 的文件重定向到 file。             |
| n >> file       | 将文件描述符为 n 的文件以追加的方式重定向到 file。 |
| n >& m          | 将输出文件 m 和 n 合并。                           |
| n <& m          | 将输入文件 m 和 n 合并。                           |
| << tag          | 将开始标记 tag 和结束标记 tag 之间的内容作为输入。 |

### 进程调度算法

Linux的进程调度算法负责决定哪个进程应该在某个时刻运行。Linux内核使用**多级反馈队列调度算法**（Multilevel Feedback Queue Scheduling Algorithm）和不同的调度策略来处理不同类型的进程。以下是Linux进程调度的核心概念和主要调度算法。

### 1. **调度策略**（Scheduling Policies）

Linux 提供了几种不同的调度策略，用于处理不同类型的进程和任务：

#### 1.1 **CFS（Completely Fair Scheduler）**

CFS 是 Linux 默认的调度算法，旨在确保系统中的每个进程公平地获得 CPU 时间。CFS 主要用于处理 **普通的用户进程**，并实现了负责任的时间共享。

- **基本思想：** CFS 基于虚拟时间（Virtual Runtime）来调度进程。每个进程都有一个虚拟运行时间，表示进程已经运行的时间。CFS 会尽量让所有进程的虚拟运行时间尽可能地公平地分配。
- 实现方式：
  - CFS 使用红黑树（Red-Black Tree）数据结构，红黑树根据每个进程的虚拟运行时间排序。虚拟运行时间越小的进程越早被调度执行。
  - 每个进程的虚拟运行时间是基于它所使用的 CPU 时间来计算的，CFS 调度器会尽量避免某个进程长时间占用 CPU。
- 优点：
  - 动态调整，确保每个进程公平地获取 CPU 时间，适合大多数交互式和批处理的应用。
  - 负载均衡表现良好，适用于多核和多处理器系统。

#### 1.2 **实时调度策略**

实时调度策略用于需要精确定时或低延迟响应的任务，通常是针对 **实时进程** 或 **嵌入式系统**。

- **SCHED_FIFO（First In First Out）**：
  - 这是最简单的实时调度策略，按照进程的到达顺序执行，优先级较高的进程会抢占较低优先级的进程。没有时间片，进程会一直运行直到它主动放弃CPU（如调用`sleep`）。
- **SCHED_RR（Round Robin）**：
  - 类似于 `SCHED_FIFO`，但每个进程都有一个时间片。当一个进程的时间片耗尽时，它会被放回队列的末尾，然后调度下一个同等优先级的进程。时间片用尽前，进程不会被抢占。
- **SCHED_DEADLINE**：
  - 这是 Linux 中用于支持实时任务的调度策略之一，基于 **Earliest Deadline First (EDF)** 算法，主要用于支持具有严格截止时间要求的实时任务。

#### 1.3 **SCHED_IDLE**

- **SCHED_IDLE** 是一个非常低优先级的调度策略。它用来调度那些非常空闲的进程或者系统中完全没有其他需要处理的任务。它的目的是让 CPU 在空闲时尽量少消耗电量。

### 2. **调度队列与优先级**

Linux 调度器使用 **多级反馈队列**（Multilevel Feedback Queue）来管理不同类型的进程。不同的队列有不同的优先级，并且调度器根据优先级来决定哪个进程应当被执行。

#### 2.1 **调度优先级**

Linux 调度器基于 **优先级** 来确定哪个进程应该被调度运行。优先级通常由两部分组成：

- **静态优先级**：表示进程的基本优先级，通常由进程的调度策略和相关的属性（如 NICE 值）决定。
- **动态优先级**：表示进程的运行状态。CFS 使用虚拟时间来动态计算进程的权重，这有助于保持系统中的公平性。

#### 2.2 **进程队列与时间片**

- **CFS队列**：CFS 调度器通过红黑树维护一个按照虚拟运行时间排序的队列，进程在队列中的位置决定了它们的调度顺序。
- **实时队列**：实时任务使用 FIFO 或 RR 调度策略，这些任务会在实时队列中优先调度，通常它们具有较高的优先级。
- **其他队列**：还有一些特殊的队列，比如用来处理 IDLE 或低优先级任务的队列。

### 3. **时间片（Time Slice）**

时间片是分配给进程的 CPU 时间。在 Linux 中，时间片是调度过程中的一个关键概念，尤其对于 **时间共享的进程** 来说至关重要。

- CFS 不使用固定的时间片，它依据进程的优先级和虚拟运行时间来动态分配时间。
- 实时调度策略（如 SCHED_FIFO 和 SCHED_RR）则有明确的时间片设定，当时间片耗尽时，进程会被抢占。

### 4. **进程调度过程**

Linux 调度器的工作流程大致如下：

1. **调度事件**：调度器被唤醒的时刻通常包括以下几种情况：

   进程的时间片耗尽。
   
   当前进程因阻塞或等待某些资源被挂起。

   进程主动放弃 CPU。

   中断、I/O 完成等。

2. **选择进程**：调度器根据当前进程的优先级、状态和调度策略，选择下一个要执行的进程。

   - 如果是实时进程，调度器会选择优先级最高的实时进程。
   - 对于普通进程（CFS），调度器选择虚拟运行时间最小的进程执行。

3. **上下文切换**：当调度器选择了一个新的进程时，会执行上下文切换，将当前进程的状态保存，并加载新进程的状态。

### 5. **调度器的负载均衡**

在多核处理器上，Linux 会进行 **负载均衡**，确保每个 CPU 核心上的负载尽可能均匀，避免某些核心过载而其他核心空闲。

- **负载均衡算法**：定期检查不同 CPU 核心上的进程负载，并将进程迁移到空闲的核心上。

### 虚拟内存

虚拟内存是操作系统的一种内存管理技术，它通过将硬盘的一部分空间作为扩展内存，使得程序可以使用比物理内存更大的内存空间。操作系统将虚拟地址映射到物理内存地址，自动管理内存的分配和回收。通过分页或分段机制，虚拟内存允许程序独立于物理内存运行，增强了系统的稳定性和效率，同时提升了多任务处理的能力。

### 页表机制

为了维护虚拟内存与物理内存之间的映射关系，操作系统使用 **页表**。页表记录了每个虚拟页与物理页框的对应关系。每个进程都有自己的页表，页表可以存储在内存中。

#### 页表的查询过程

当 CPU 要访问某个虚拟地址时，首先通过 **MMU（内存管理单元）** 查找虚拟地址的页表项，进行虚拟地址到物理地址的转换。这个过程通常包括以下几个步骤：

1. **获取虚拟地址**
   - 分割虚拟地址为虚拟页号和页内偏移。
2. **查找 TLB**
   - 如果 TLB 命中，直接返回物理地址。
   - 如果 TLB miss，继续查找页表。
3. **查找页表**
   - 查找虚拟页号对应的物理页框号。
   - 将物理页框号与页内偏移结合，得到物理地址。
4. **更新 TLB**
   - 将新映射存入 TLB。
5. **计算物理地址**
   - 使用物理页框号和页内偏移计算物理地址，完成内存访问。

### 多级页表（Multi-level Page Table）

当虚拟地址空间较大时，单级页表可能变得非常庞大，导致内存浪费。为了节省内存，现代操作系统常使用 **多级页表**。多级页表通过分层来管理虚拟地址与物理地址的映射，每一级页表存储的是下一级页表的地址，而不是直接存储物理页框号。

### 换页

换页是虚拟内存管理中的一个重要机制。当程序请求访问的页面不在物理内存中时，操作系统会从硬盘中将该页面加载到内存中，这个过程称为换页。换页使得程序能够使用比物理内存更大的虚拟内存，增强了内存的使用效率。虽然换页机制提高了内存利用率，但频繁的换页会导致性能下降，因为硬盘的读写速度远低于内存。换页通常伴随有缺页中断和页替换算法的管理。

**换页分类**：

1. **缺页中断（Page Fault）**：当访问的页面不在内存中时，触发缺页中断，操作系统会从磁盘加载页面。
2. **脏页换入（Dirty Page Fault）**：当一个页面在内存中被修改后，还未写回磁盘，发生换入时需要先将脏页保存到磁盘。

**抖动（Thrashing）**：

- 当内存不足且频繁发生页面交换时，操作系统不断将页面从内存换到硬盘，再从硬盘换回内存，导致大量的磁盘I/O操作，严重拖慢系统性能。这种现象被称为“抖动”。
- 为了避免抖动，操作系统会尽量减少换页次数，采用页面替换算法来决定哪些页面应该留在内存中，哪些页面可以交换出去。

**页面替换（Page Replacement）**：

- 如果内存已满，操作系统必须选择一个页面进行替换。常见的页面替换算法有：
  - **最少使用（LFU）**：替换那些最久未被访问的页面。
  - **最近最少使用（LRU）**：替换最近最少使用的页面。
  - **先进先出（FIFO）**：按照页面加载的先后顺序替换最早加载的页面换页

**`vm.page-cluster`** 一次交换的页数

### linux过度提交

- **共享内存资源**：并非所有进程或虚拟机同时需要最大量的内存。例如，多个虚拟机可能在不同的时间点使用不同的内存量，实际的内存需求并不完全重叠。
- **内存页共享和交换**：操作系统可以通过共享内存页、内存压缩、交换空间等手段，在内存资源不足时进行动态调整。

Linux通过虚拟内存管理机制实现过度提交。当进程请求内存时，操作系统并不立即分配物理内存，而是分配虚拟内存地址空间，实际物理内存可能不足。内核会根据过度提交策略（如always、never、heuristic）决定是否允许过度提交。如果启用过度提交，内核会在物理内存不足时使用交换空间（swap）或回收内存页，甚至杀死占用内存过多的进程（OOM）。这种机制依赖于内存管理、交换机制和内核策略。

过度提交相关参数 :**`vm.overcommit_memory`**     控制内存过度提交的策略（0，1，2）->0 交由系统绝对 1始终可以 2始终不可以

**`vm.overcommit_ratio`** 过度提交内存的比例设置

**`vm.overcommit_kill`**  oom情况下 杀死占用过多的进程

### Cpu缓存

CPU缓存的核心作用是减少CPU对主内存（RAM）进行访问的次数，因为RAM相对较慢，访问延迟较大。通过缓存，CPU可以更快地从缓存中获取常用的数据和指令，而不是每次都从更慢的主内存中读取。

大概分为3级缓存 L1<L2<L3 但缓存速度 相反

原理主要是:局部性原理

- **缓存命中**：如果数据已在缓存中，CPU直接访问缓存。
- **缓存未命中**：数据不在缓存中，CPU从内存加载并更新缓存。



### 文件系统预取（File System Prefetching）

是一种通过预测应用程序或用户可能访问的数据并提前将其加载到内存中来优化磁盘I/O性能的技术。预取机制的核心目的是减少磁盘访问的延迟，提升系统的整体响应速度。

通过这些局部性原则，文件系统能够根据过去的行为推测出将来可能访问的文件块。 预先读取可以提高效率

#### **监测访问模式**

操作系统监控程序对磁盘文件的访问模式，分析出它们的访问规律。这种监控可以依赖于文件系统本身，也可以通过内核提供的缓存来实现。常见的监测模式包括：

- **顺序访问**：例如，大多数情况下，用户会顺序读取一个文件，或者按块读取一个数据库。
- **随机访问**：一些程序可能会随机访问数据块，例如数据库查询、非顺序文件访问等。

#### 2. **预测未来的访问**

基于对历史访问数据的分析，文件系统会**预测**哪些数据块可能会在未来被访问。预测的方法有很多种，最常见的包括：

- **顺序访问预测**：如果应用程序读取文件的一部分，那么它很可能会继续读取文件的后续部分。比如，如果读取了一个文件的前1024字节，系统可以预测应用程序可能还会访问后1024字节，提前加载到内存中。
- **跳跃性访问预测**：如果程序跳着访问某些特定数据块，文件系统可能会猜测接下来会访问相似的其他数据块。比如，一些数据库查询可能会有特定的访问模式，文件系统会根据这个模式提前加载相关数据。

#### 3. **数据预取**

当文件系统预测到某些数据块将被访问时，它会主动去磁盘读取这些数据，并将它们加载到内存中。这一过程通常是异步的，也就是说，文件系统在后台执行这些预取操作，不会阻塞用户程序的运行。

- **异步预取**：文件系统在用户程序进行文件访问时，不会等待数据从磁盘加载完成，而是提前将数据加载到内存中，这样程序在需要访问这些数据时就可以直接从内存中读取，避免了磁盘访问延迟。

#### 4. **缓存管理**

加载的数据存放在操作系统的缓存（例如页面缓存）中。操作系统通常会使用一些缓存替换算法（如**LRU（Least Recently Used）**）来管理这些数据。当缓存满时，系统会根据预定的策略清除旧的数据，确保常用的数据保持在内存中。

#### 优点

- **减少I/O延迟**：预取能够有效降低磁盘的访问延迟，尤其是在读取大文件或连续数据时，提升程序的响应速度。
- **提升吞吐量**：通过减少磁盘访问次数，预取可以提高磁盘带宽的利用率，尤其是在需要顺序读取大量数据的应用中。

#### 缺点

- **内存浪费**：如果预取策略不准确，可能会将一些不必要的数据加载到内存中，导致内存资源浪费。
- **预取失效**：对于访问模式非常随机的应用，预取策略可能会导致性能下降，因为它预加载的数据可能不会被实际访问。
- **增加系统负担**：过度的预取可能导致系统在后台进行大量的磁盘I/O操作，增加系统的负载，特别是在资源紧张的环境下



###      1. **同步I/O（Synchronous I/O）**

同步I/O是指在进行I/O操作时，调用方（如进程或线程）会被阻塞，直到I/O操作完成后才会继续执行。简单来说，执行I/O操作的线程会等待操作完成后再继续处理其他任务。

####       工作原理：

- 当程序发起I/O请求时（比如读取文件或向文件写数据），操作系统会将请求发送到硬件设备或文件系统。

- 直到I/O操作完成，操作系统才会通知程序。

- 在此期间，程序会处于阻塞状态，不能执行其他任务，直到I/O操作完成。

  ```python
  # Python示例：同步I/O
  file = open('data.txt', 'r')
  data = file.read()  # 直到文件读取完毕，程序才会继续执行
  file.close()
  ```

  

  ### 2. **异步I/O（Asynchronous I/O）**

  异步I/O是指程序发起I/O操作时，不会阻塞程序的执行。程序会继续执行其他任务，直到I/O操作完成后，通过回调、事件或通知机制来处理I/O操作的结果。

  #### 工作原理：
  
  - 当程序发起I/O请求时，操作系统将请求交给后台处理，程序本身不会被阻塞。
  - 程序可以继续执行其他计算任务，直到I/O操作完成。
  - 一旦I/O操作完成，操作系统通过回调函数或事件机制通知程序结果。
  
  ```python
  import asyncio
  
  async def read_file():
      with open('data.txt', 'r') as file:
          data = await file.read()  # 异步读取，程序不会被阻塞
          print(data)
  
  asyncio.run(read_file())  # 运行异步任务
  ```

### AIO使用流程

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <fcntl.h>
#include <sys/types.h>
#include <sys/stat.h>
#include <aio.h>
#include <string.h>

#define FILE_NAME "testfile.txt"
#define BUFFER_SIZE 1024

int main() {
    int fd;
    struct aiocb aio;
    char buffer[BUFFER_SIZE];

    // 打开文件
    fd = open(FILE_NAME, O_RDONLY);
    if (fd == -1) {
        perror("open");
        exit(EXIT_FAILURE);
    }

    // 设置 AIO 控制块 初始化
    memset(&aio, 0, sizeof(struct aiocb));
    aio.aio_fildes = fd;
    aio.aio_buf = buffer;
    aio.aio_nbytes = BUFFER_SIZE;
    aio.aio_offset = 0;

    // 提交异步读操作 读取
    if (aio_read(&aio) == -1) {
        perror("aio_read");
        close(fd);
        exit(EXIT_FAILURE);
    }

    // 等待操作完成
    while (aio_error(&aio) == EINPROGRESS) {
        // 可以执行其他任务
    }

    // 检查操作是否成功
    if (aio_error(&aio) != 0) {
        perror("aio_error");
        close(fd);
        exit(EXIT_FAILURE);
    }

    // 获取操作结果
    ssize_t bytes_read = aio_return(&aio);
    printf("Read %zd bytes: %s\n", bytes_read, buffer);

    // 关闭文件
    close(fd);
    return 0;
}

```

### barrier 屏障

屏障（Barrier）是一种同步机制，常用于多线程编程中，确保多个线程在继续执行之前都到达某个同步点。屏障可以帮助协调线程的执行顺序，通常用于需要阶段性同步的场景。

##### 屏障的工作原理

1. **计数器**：屏障通常维护一个计数器，用于记录已经到达屏障的线程数量。

2. **等待机制**：当一个线程到达屏障时，它会增加计数器的值并检查是否所有线程都已到达。如果不是，它将进入等待状态。

3. **释放机制**：当最后一个线程到达屏障时，计数器达到预期值，所有等待的线程被唤醒并继续执行。

### 1. 指令流水线 (Instruction Pipeline)

指令流水线是一种将指令执行过程划分为多个阶段的技术，使得多个指令能够在不同的阶段并行执行。每个指令的执行被分解成多个阶段，例如取指阶段、解码阶段、执行阶段、内存访问阶段、写回阶段等。每个阶段都由不同的硬件单元并行处理，不同指令在流水线的不同阶段上运行。

#### **为什么要**

- **提高CPU的吞吐量**：通过并行处理多条指令，不仅可以减少每条指令的平均执行时间，还能提高CPU的整体执行效率。
- **减少等待时间**：由于每个指令只在某个特定阶段执行，CPU不会等待某个指令完成后再开始下一条指令的执行。
- **最大化硬件资源利用**：硬件的不同部分可以同时执行不同的任务，从而提高资源利用率。

实现指令流水线需要硬件支持，通常通过以下方式：

- **分段设计**：将指令的执行过程分为多个阶段，如取指、解码、执行、访存、写回等。
- **并行处理**：每个阶段由不同的硬件单元并行处理不同指令的相应部分。
- **流水线深度**：流水线的深度决定了每条指令通过流水线的阶段数，深度越大，处理速度越快，但复杂度也会增加。

流水线技术还涉及到一些优化，例如：

- **数据冒险**：当一个指令的执行依赖于前一个指令的结果时，可能会导致数据冒险。通过技术如数据前递和插入等待周期来解决。
- **控制冒险**：分支指令会改变程序流，可能导致流水线中的指令无法按预期顺序执行。通过分支预测等技术解决。

### 2. 线程 (Thread)

#### **是什么**

线程是程序中的一个执行单元，是 CPU 调度和执行的基本单位。每个线程都是在进程内执行的，多个线程共享同一进程的内存空间和资源，但它们有独立的执行路径。线程可以并发执行，从而提高程序的效率，特别是在多核处理器的环境下。

#### **为什么要**

- **提高程序并行性**：多个线程可以并行执行，从而减少整体的执行时间，特别是在 I/O 操作、计算密集型任务等情况下。
- **优化资源使用**：多个线程共享进程的内存和资源，避免了多进程带来的资源浪费。
- **提高响应性**：多线程可以使得程序在等待某个任务时继续执行其他任务，提高系统的响应速度（如在 GUI 应用中，主线程可以处理用户界面，而子线程处理耗时操作）。

#### **如何实现**

线程的实现通常依赖于操作系统的线程管理机制，主要方式有：

- **用户级线程**：线程的管理由用户空间的线程库（如 pthread）实现，操作系统不直接参与线程的管理。这种方式的优点是线程创建和销毁的开销小，但如果一个线程阻塞，整个进程都可能被阻塞。
- **内核级线程**：操作系统直接管理线程，操作系统内核负责线程的创建、销毁、调度和同步。每个线程都有独立的栈和控制块，内核能够直接对线程进行调度。这种方式适合多核处理器，可以更好地支持并发。
- **混合模式**：结合用户级线程和内核级线程，通常由用户空间线程库和内核同时管理线程。

线程的调度通常由操作系统根据某些策略（如时间片轮转、优先级调度等）来分配 CPU 资源。

### 3. 抢占 (Preemption)

#### **是什么**

抢占是指操作系统在多任务环境下强制将 CPU 从一个正在运行的进程或线程中夺走，转而将其分配给另一个进程或线程。这通常是由于时间片耗尽、优先级更高的任务到达等原因，操作系统会主动中断当前执行的任务，以保证系统的响应性和公平性。

#### **为什么要**

- **提高响应性**：抢占可以保证高优先级的任务及时得到执行，提升系统对外部事件（如用户输入、I/O 请求等）的响应能力。
- **增强系统公平性**：抢占式调度能够防止某个低优先级任务占用过多 CPU 时间，导致其他任务无法执行。
- **提高并发性**：抢占能够有效利用多核处理器资源，保证多个任务并发执行，充分发挥多任务操作系统的优势。

#### **如何实现**

抢占的实现依赖于操作系统的调度机制，通常通过以下方式：

- **定时器中断**：操作系统定期设置定时器中断，当一个任务的时间片用完时，操作系统会强制中断当前任务的执行，转而调度另一个任务。
- **优先级调度**：操作系统可以基于任务的优先级来决定是否进行抢占，如果一个更高优先级的任务需要 CPU，操作系统会抢占当前任务。
- **抢占式多任务操作系统**：现代操作系统（如 Linux、Windows 等）都采用抢占式调度，意味着操作系统可以随时中断当前正在运行的任务，将 CPU 分配给其他任务。

抢占式调度需要保证任务状态的保存和恢复（上下文切换），操作系统会保存当前任务的上下文（寄存器状态、堆栈等），并在抢占后恢复被抢占任务的执行。

`bpftrace` 是一个强大的动态跟踪工具，基于 BPF（Berkeley Packet Filter），能够通过事件跟踪、系统调用、函数调用等方式提供对内核和用户空间的高效分析。编写 `bpftrace` 脚本时，需要遵循一些规则和语法结构，确保脚本能够正确运行。

### 1. **BPFTrace 脚本的基本结构**

`bpftrace` 脚本通常由以下几个部分组成：

- **声明**：指定脚本的执行上下文、事件等。
- **事件监听器**：指定需要跟踪的内核或用户空间事件（如函数、系统调用、tracepoint 等）。
- **操作**：定义如何处理捕获的事件，通常是打印、统计、计算等。
- **条件判断和控制结构**：如 if-else、循环等，用于精细控制脚本的行为。

### 2. **BPFTrace 脚本的语法规则**

#### 事件监听器（Tracing）

事件监听器用于指定脚本要跟踪的特定事件或 tracepoint。常见的事件有：

- **tracepoint**：例如，`tracepoint:syscalls:sys_enter_read` 监听 `sys_enter_read` 事件。
- **kprobe**：跟踪内核函数入口，例如，`kprobe:sys_open` 跟踪内核 `sys_open` 函数的调用。
- **uprobe**：跟踪用户空间函数入口。
- **profile**：周期性地捕获事件，通常用于 CPU 采样。
- **profile:hz**：每个时钟周期（通常是每秒的次数）跟踪一个事件。

#### 示例：

```
tracepoint:syscalls:sys_enter_read
{
    printf("Read system call entered\n");
}
```

这个脚本监听 `syscalls:sys_enter_read` 事件，每次发生时打印一条信息。

#### 变量和数据存储

- **局部变量**：用来存储简单的数据。局部变量在事件处理过程中有效，通常在 `{}` 内定义。

  示例：

  ```
  tracepoint:syscalls:sys_enter_read
  {
      pid = pid;  // 存储 pid
  }
  ```

- **关联数组**（Associative arrays）：可以根据某个值（如进程 ID）索引存储数据，通常用于统计或跟踪跨事件的数据。

  示例：

  ```
  tracepoint:syscalls:sys_enter_read
  {
      @reads[pid] = count();  // 使用进程 ID 索引，记录每个进程的读取次数
  }
  ```

- **全局数组**：可以在多个事件中共享数据（如全局统计）。

  示例：

  ```
  tracepoint:syscalls:sys_enter_read
  {
      @read_count = count();
  }
  ```

#### 打印和输出

- **printf**：用于格式化输出。类似于 C 语言中的 `printf`，支持格式化字符串和变量。

  示例：

  ```
  tracepoint:syscalls:sys_enter_read
  {
      printf("Process %d is reading\n", pid);
  }
  ```

#### 计算和条件

BPFTrace 支持数学运算、条件判断、逻辑操作等。

- **算术运算**：`+`, `-`, `*`, `/` 等。

  示例：

  ```
  tracepoint:syscalls:sys_enter_read
  {
      @total_bytes = @total_bytes + args->count;
  }
  ```

- **条件判断**：`if`、`else` 语句，允许根据条件决定如何处理事件。

  示例：

  ```
  tracepoint:syscalls:sys_enter_read
  {
      if (args->count > 1024) {
          printf("Large read detected\n");
      }
  }
  ```

### 3. **常用操作**

#### 变量声明和使用

- **标量变量**：标量变量用于存储简单的单一值。通过变量名引用。

  ```
  tracepoint:syscalls:sys_enter_read
  {
      @total_read_bytes = @total_read_bytes + args->count;
  }
  ```

- **关联数组**：数组类型变量，常用于通过 PID、进程名称、文件描述符等标识来跟踪事件数据。

  ```
  tracepoint:syscalls:sys_enter_read
  {
      @read_count[pid] = count();
  }
  ```

#### 删除变量

- **删除数组中的元素**：可以使用 `delete` 命令删除某个数组项。

  ```
  tracepoint:syscalls:sys_exit_read
  {
      delete(@read_count[pid]);
  }
  ```

### 4. **控制结构**

BPFTrace 脚本支持常见的控制结构，例如条件判断和循环。

- **条件判断**：`if`、`else`。

  ```
  tracepoint:syscalls:sys_enter_read
  {
      if (args->count > 1024) {
          printf("Large read\n");
      }
  }
  ```

- **循环**：支持对数组进行遍历或进行一定次数的循环操作。

  示例：

  ```
  interval:s:1
  {
      foreach (pid in @read_count) {
          printf("PID %d read count: %d\n", pid, @read_count[pid]);
      }
  }
  ```

### 5. **聚合和统计**

- **`count()`**：计算事件发生的次数。
- **`sum()`**：计算某个值的总和。
- **`avg()`**：计算某个值的平均值。
- **`min()` 和 `max()`**：计算某个值的最小值和最大值。

例如，统计进程 I/O 请求的次数：

```
tracepoint:block:block_rq_issue
{
    @rq_count[pid] = count();
}
```

### 6. **退出和调试**

- **退出脚本**：可以使用 `exit` 来终止脚本执行。

  示例：

  ```
  tracepoint:syscalls:sys_enter_read
  {
      printf("Exiting after one read\n");
      exit();
  }
  ```

### 7. **执行环境和作用域**

- **局部作用域**：在某个事件块内部定义的变量是局部的，仅在该事件的处理过程中有效。
- **全局作用域**：以 `@` 开头的变量是全局的，跨多个事件使用。

### 8. **性能注意事项**

虽然 `bpftrace` 提供了强大的功能，但由于它是动态地在内核中插入钩子来跟踪事件，过多的 `printf` 输出、复杂的计算和过度使用全局数组可能影响系统性能，特别是在高频事件（如系统调用、网络包等）上运行时。

### c2c

**C2C** 是 **Cache-to-Cache**（缓存到缓存）性能分析的缩写，通常用于描述在多核或多处理器系统中，**一个处理器的缓存与另一个处理器的缓存之间的数据交换**。这种缓存交互是多核处理器架构中提升性能的一个关键部分，尤其是当多个核心需要共享内存或缓存时。C2C 主要关注的是**不同处理器或核心之间的数据一致性、缓存命中和缓存未命中的情况**。

### 1. **C2C 的背景和意义**

在多核系统中，每个 CPU 核心通常都会有自己独立的缓存（L1 和 L2 缓存，甚至是 L3 缓存）。为了提高性能，CPU 设计通常会试图将访问的数据保留在局部缓存中，而不是每次都从主内存（RAM）中读取。这样，缓存命中会显著加速数据访问。

然而，当多个核心访问相同的数据时，可能会涉及到跨核心之间的数据传输。比如，核心 1 修改了某个缓存行，核心 2 可能需要从核心 1 的缓存中读取该数据。这个过程就是 **Cache-to-Cache（C2C）** 传输。

C2C 主要用于优化和分析跨核心、跨处理器之间的数据传输与一致性。它帮助我们理解在多核系统中，缓存如何在核心之间共享，哪些数据经常在不同的核心间传输，以及如何减少这类跨缓存访问以提高整体系统性能。

### 2. **C2C 的典型操作**

在多核处理器中，C2C 操作通常会涉及以下几种情况：

- **缓存一致性协议（Cache Coherence Protocols）**：为确保系统中的所有 CPU 核心都能看到数据的一致视图，现代多核处理器实现了缓存一致性协议（如 MESI 协议，Modified, Exclusive, Shared, Invalid）。这个协议确保了当一个核心修改某个缓存行的数据时，其他核心能够知道这个变化，避免出现数据不一致的问题。
- **缓存行共享**：当多个核心同时访问同一缓存行时，数据可能被复制到不同核心的缓存中。这种共享的缓存行会在不同核心之间发生传输，C2C 通过追踪这种缓存行的共享行为来优化性能。
- **远程缓存访问**：如果核心 A 想要读取核心 B 缓存中的数据，那么就需要进行跨核心（C2C）访问。这时会产生一定的延迟，因为数据可能需要从远程核心的缓存中拉取（而非本地缓存）。

### 3. **C2C 性能分析的目标**

在进行 C2C 性能分析时，通常的目标包括：

- **检测缓存未命中（Cache Miss）**：分析跨核心的缓存未命中现象，即某个核心试图访问另一个核心缓存中的数据，导致延迟。这些跨核心的访问可能会显著影响程序性能，特别是当缓存一致性协议频繁介入时。
- **减少远程访问（Remote Access）**：分析不同核心之间的缓存访问模式，尽量减少远程缓存访问，尤其是从其他 CPU 核心读取缓存行的情况。远程访问通常比本地访问慢，因为它涉及到跨 CPU 核心和/或跨 NUMA 节点的数据传输。
- **提高数据局部性**：通过分析 C2C 交互，可以发现哪些数据在不同核心之间频繁移动。这些数据通常是共享数据，优化这些数据的访问模式有助于提升性能。
- **优化数据共享**：通过有效的缓存共享，可以减少跨核心的缓存传输，从而提高程序的并行性和处理能力。

### 4. **C2C 分析的常见指标**

在进行 C2C 性能分析时，常常会关注以下一些重要的指标：

- **本地缓存命中率（Local Cache Hits）**：本地核心缓存中直接找到数据的比例。较高的命中率可以减少跨核心的缓存访问，从而提升性能。
- **远程缓存命中率（Remote Cache Hits）**：其他核心缓存中的命中比例。较高的远程命中率通常表明缓存一致性协议比较活跃，数据频繁跨核心移动。
- **跨缓存传输（Cross-cache Transfers）**：这通常指一个核心必须从另一个核心的缓存中读取数据。跨缓存传输的次数越多，意味着数据共享的开销越大，可能会导致性能下降。
- **NUMA 访问模式**：如果系统使用 NUMA（非一致性内存访问）架构，跨 NUMA 节点的缓存访问将变得更加复杂和昂贵。C2C 分析可以帮助优化跨节点的数据传输。

### bitmap

`bitmap_alloc` 是一个专门用于分配和初始化位图（bitmap）内存的函数。位图是一种高效的存储结构，用于表示一组布尔值，通常用于追踪一组资源的状态或某些集合中的元素。

### 1. **位图（Bitmap）简介**

位图是一种紧凑的数据结构，它将每个元素的状态（存在/不存在、开启/关闭等）表示为一个位（0 或 1）。它通常用于处理大量布尔数据，并提供高效的空间利用和快速的查询、更新操作。位图常见的应用场景包括：

- 标记和追踪集合中的元素。
- 管理资源池（如 CPU 核心、内存页、硬件设备等）。  -->perf中使用了
- 实现集合的并集、交集、差集等操作。

例如，一个包含 100 个元素的集合，如果用位图表示，实际上只需要 100 个二进制位，表示每个元素是否存在。

### 2. **`bitmap_alloc` 函数**

`bitmap_alloc` 通常是一个用于动态分配内存并初始化为零的函数，专门为位图数据结构设计。它的作用是分配一块内存，用于存储指定大小的位图，并确保每一位初始化为 0（即布尔值“假”或“未分配”状态）。

#### 语法：

```c

unsigned long *bitmap_alloc(int bits);
```

- **参数**：
  - `bits`：要分配的位图大小，以位（bit）为单位。通常为所需位图元素的数量。例如，如果你想要一个大小为 100 位的位图，`bits` 就应该是 100。
- **返回值**：
  - 返回一个指向分配的位图内存的指针，内存中的每一位都会初始化为 0。
  - 如果内存分配失败，则返回 `NULL`。

#### 3. **如何实现位图**

通常，位图会存储在一个数组或一个长整型数（`unsigned long`）数组中。每个数组元素用于存储多个二进制位，通常一个 `unsigned long` 可以容纳 64 位。在这种情况下，位图中的每一位将用于表示一个布尔值。

例如，假设我们需要表示 100 个布尔值，位图的长度将是 100/64 = 2（两个 `unsigned long` 数组元素）。每个数组元素会存储 64 个布尔值，因此实际上这个位图包含 128 位，但只会使用前 100 位。

### 4. **位图的内存分配**

由于位图实际上是对多个比特位的管理，内存分配通常是基于一个长整型数组或类似的结构。每个 `unsigned long` 类型的变量能够存储 64 位（通常是 64 位操作系统下的机器字长），因此 `bitmap_alloc` 会分配适当数量的 `unsigned long` 数组，以容纳指定数量的位。

#### 内存分配示例：

假设我们需要分配一个大小为 100 位的位图：

1. 计算需要的 `unsigned long` 数组的数量：`100 / 64 = 2`（即需要两个 `unsigned long` 来存储 100 位）。
2. 分配两个 `unsigned long` 的内存空间，保证能存储 100 位。

### 5. **常见实现**

一个简单的 `bitmap_alloc` 实现可能如下所示：

```c
#include <stdlib.h>
#include <string.h>

unsigned long *bitmap_alloc(int bits) {
    int size = (bits + 63) / 64;  // 计算所需的 unsigned long 数组数量
    unsigned long *bitmap = (unsigned long *)malloc(size * sizeof(unsigned long));
    if (bitmap) {
        memset(bitmap, 0, size * sizeof(unsigned long));  // 初始化为 0
    }
    return bitmap;
}
```

#### 解释：

- `bits + 63) / 64`：计算需要多少个 `unsigned long` 来存储指定的位数。每个 `unsigned long` 可以存储 64 位，因此需要的数组大小是 `(bits + 63) / 64`。
- `malloc`：分配内存块来存储位图。
- `memset`：将内存中的所有字节设置为 `0`，以确保位图的每一位都初始化为 `0`。

### 6. **如何使用位图**

位图的使用主要涉及以下几个操作：

- **设置某一位**：将位图的指定位置的值设置为 1（通常表示某个资源被分配或某个条件满足）。
- **清除某一位**：将位图的指定位置的值设置为 0（通常表示某个资源被释放或条件不再满足）。
- **检查某一位的状态**：查看位图某一位的值，用于判断某个资源是否处于某个状态。

常见的位图操作：

```c
#include <stdio.h>
#include <stdlib.h>

// 设置某个位置的位为 1
void bitmap_set(unsigned long *bitmap, int bit) {
    bitmap[bit / 64] |= (1UL << (bit % 64));  // 设置指定位置的位
}

// 清除某个位置的位为 0
void bitmap_clear(unsigned long *bitmap, int bit) {
    bitmap[bit / 64] &= ~(1UL << (bit % 64));  // 清除指定位置的位
}

// 检查某个位置的位是否为 1
int bitmap_test(unsigned long *bitmap, int bit) {
    return (bitmap[bit / 64] & (1UL << (bit % 64))) != 0;  // 判断指定位置的位是否为 1
}

int main() {
    int bits = 100;
    unsigned long *bitmap = bitmap_alloc(bits);  // 分配一个大小为 100 位的位图

    if (bitmap) {
        // 设置第 5 位
        bitmap_set(bitmap, 5);
        // 检查第 5 位
        printf("Bit 5 is %d\n", bitmap_test(bitmap, 5));  // 输出 1，表示该位置为 1
        // 清除第 5 位
        bitmap_clear(bitmap, 5);
        // 检查第 5 位
        printf("Bit 5 is %d\n", bitmap_test(bitmap, 5));  // 输出 0，表示该位置为 0

        free(bitmap);  // 释放内存
    }

    return 0;
}
```

### 7. **优点**

- **高效的空间利用**：位图只使用一个位来表示每个布尔值，节省了内存空间。对于大规模的集合或资源池，位图可以比传统的数据结构（如数组或链表）节省大量空间。
- **快速的访问**：位图支持 O(1) 的查询、设置和清除操作，可以非常快速地执行。

### 8. **缺点**

- **固定大小**：位图的大小是固定的，因此如果集合的大小发生变化，可能需要重新分配和调整位图的大小。
- **浪费内存**：如果位图的大小远大于实际使用的元素数量（例如，分配了 1000 位，但只使用了 10 位），会浪费内存。

### Snoop 协议详解

**Snoop 协议**（Snooping Protocol）是现代计算机体系结构中一种用于 **缓存一致性** 的机制，特别在 **多核** 或 **多处理器** 系统中，帮助维持不同处理单元之间的一致性。简单来说，Snoop 协议允许处理器之间通过监视（或“窥探”）其他处理器的缓存来确保内存中的数据一致。

当多个处理器或多个核心共享内存时，它们各自可能有自己的本地缓存（L1、L2 缓存等）。为了避免缓存数据不同步的问题，需要通过某种协议来确保内存数据的 **一致性**。

#### 1. **缓存一致性问题**

在多处理器系统中，每个处理器都有自己的缓存。如果一个处理器修改了缓存中的某个数据，其他处理器如果没有得到及时的更新，它们可能继续读取到过时的数据。为了解决这个问题，缓存一致性协议应运而生。

主要问题包括：

- **写缓存**：处理器对缓存的写操作可能不立即同步到内存或其他处理器的缓存。
- **读取缓存**：其他处理器可能继续读取到过时的数据。

#### 2. **Snoop 协议的工作原理**

在 Snoop 协议中，每个处理器的缓存通过网络与其他处理器的缓存进行通信。每当一个处理器对缓存中的数据进行修改时，其他处理器的缓存会“窥探”这些修改，从而保证它们的数据保持一致。

Snoop 协议的基本工作流程通常包括：

- **监控缓存操作**：每个处理器的缓存控制器都能监听其他处理器对内存或缓存的操作，尤其是对共享数据的访问。
- **缓存更新**：当某个处理器对缓存中的数据进行修改时，其他处理器通过监听（snoop）该操作，并根据需要更新它们的缓存数据。

#### 3. **Snoop 协议的类型**

在 Snoop 协议中，主要有两种常见的协议：

- **MESI 协议**（Modified, Exclusive, Shared, Invalid）

  - **Modified (M)**: 数据在本地缓存中被修改，并且与内存中的数据不同。
  - **Exclusive (E)**: 数据仅存在于本地缓存，并且与内存中的数据一致。
  - **Shared (S)**: 数据可能在多个缓存中被共享，并且与内存中的数据一致。
  - **Invalid (I)**: 数据在缓存中是无效的，不能用于读取。

  MESI 协议是最常见的缓存一致性协议，每个缓存行有一个状态来表示它的数据是否是有效的、共享的、修改的或是独占的。

  **工作流程**：

  - 当处理器修改缓存数据时，它会将数据标记为                                                                   。
  
- **MOESI 协议**（Modified, Owned, Exclusive, Shared, Invalid） MOESI 协议是 MESI 协议的扩展，增加了一个 **Owned** 状态，表示缓存持有该数据并且数据可能被其他缓存共享，但缓存中的数据是最新的。

  **工作流程**：

  - **Owned (O)**：这个缓存行的数据是最新的，但它不一定是唯一持有该数据的缓存。其他处理器也可能有该数据的副本，但缓存会负责确保数据的一致性。

#### 4. **Snoop 协议的实现**

**Snoop 协议的实现**通常依赖于硬件层面的支持，尤其是 **缓存控制器** 和 **总线**。处理器和缓存控制器通过总线互联，当某个处理器发生缓存读写时，其他处理器的缓存会“窥探”总线上的操作。

- **总线**：多处理器系统中的总线扮演着一个关键角色，它负责将处理器的缓存操作广播给其他处理器。
- **缓存控制器**：每个处理器的缓存控制器负责监听总线上发生的操作，并决定是否需要更新或失效自己的缓存数据。

#### 5. **缓存一致性协议的优缺点**

**优点**：

- **简单性**：Snoop 协议相对简单，特别适用于共享内存系统，其中所有处理器共享相同的物理内存。
- **低延迟**：由于缓存控制器直接监听总线操作，处理器可以快速做出响应，避免了额外的内存访问延迟。

**缺点**：

- **带宽瓶颈**：由于所有的缓存操作都需要通过总线广播，随着处理器数量的增加，总线带宽可能成为瓶颈。
- **扩展性差**：Snoop 协议的扩展性有限，当处理器数量增加时，缓存同步可能会变得更加复杂，性能会受到影响。

#### 6. **Snoop 协议应用实例**

在许多现代的多核处理器体系结构中，如 **ARM**、**x86** 和 **PowerPC**，Snoop 协议都是用来保证缓存一致性的基础。以 **Intel** 的 **MESI 协议**为例：

- 当一个处理器将数据从共享状态变为修改状态时，它会通知其他处理器使其缓存中的该数据无效（Invalid）。
- 如果另一个处理器尝试读取这个被修改的数据，它会通过 Snoop 协议检测到数据被修改，并从主内存或缓存中的最新数据获取。

### 强弱符号介绍

在计算机系统，尤其是操作系统、编译器和链接器中，符号的概念非常重要。**符号**是程序中代表函数、变量或其他标识符的名称。当程序源代码编译并链接时，符号用于连接不同的代码模块和共享库。

**强符号**和**弱符号**是链接过程中的两个重要概念，它们用于处理符号在多个对象文件中可能出现重复的情况。下面详细介绍这两种符号。

#### 1. **强符号（Strong Symbol）**

**强符号**是指具有明确定义的符号，通常由程序中的某个源文件提供。在链接时，如果有多个目标文件或库中定义了同名的符号，链接器会选择其中一个定义作为最终的符号。强符号的特点是：

- **唯一性**：强符号必须有一个明确的定义，并且在整个程序中必须唯一。
- **冲突**：如果多个目标文件定义了相同的强符号，链接器会报错，提示符号冲突。程序员需要解决冲突，通常通过重命名符号或使用 `extern` 来指定外部符号。
- **优先级**：强符号具有更高的优先级，如果程序中有强符号与弱符号同名，链接器会选择强符号。

#### 示例：

```c

int global_var = 10;  // 强符号
```

#### 2. **弱符号（Weak Symbol）**

**弱符号**是指符号的定义存在，但它的优先级较低。如果链接器在多个目标文件中遇到同名的符号时，弱符号将被忽略，选择其他强符号作为最终的符号定义。弱符号主要用于以下几种情况：

- **备用定义**：弱符号常用于提供某个函数或变量的默认实现，或者给出一个占位符，在链接时可以被替换为更强的符号定义。例如，C 标准库中的许多默认函数就是弱符号。
- **可选符号**：当某个符号并非程序运行时的必要条件时，弱符号可以被使用。如果链接时没有找到任何强符号定义，弱符号会作为默认符号使用。

#### 特点：

- **默认值**：弱符号通常会赋予符号一个默认值，或提供一个默认实现。
- **冲突**：如果在链接过程中找到了一个强符号与弱符号同名，强符号会覆盖弱符号。
- **在多个目标文件中定义**：多个目标文件或库中可以同时定义同名的弱符号，链接器会选择其中一个定义，并忽略其他弱符号。

#### 示例：

```c

__attribute__((weak)) int global_var = 20;  // 弱符号
```

或者：

```c
__attribute__((weak)) void func() {
    // 默认实现
    printf("This is a weak function\n");
}
```

#### 3. **强弱符号在链接过程中的行为**

当链接器处理符号时，会遵循以下规则：

- **弱符号和强符号的选择**：如果有一个强符号和一个弱符号同名，链接器总是选择强符号。
- **多个弱符号**：如果有多个弱符号同名，链接器会选择其中一个，通常是第一个遇到的符号。
- **没有强符号时，使用弱符号**：如果没有找到强符号，链接器会选择弱符号作为默认定义。
- **多个强符号**：如果有多个强符号同名，链接器会报错，提示符号冲突。

#### 4. **常见的应用场景**

##### 4.1 **默认实现**

弱符号常用于提供某些功能的默认实现，尤其是在库开发中。例如，某些函数可以使用弱符号定义，以便用户可以在自己的程序中提供自定义实现。如果用户没有提供自定义实现，则会使用库中定义的弱符号。

例如，标准 C 库中的 `malloc` 和 `free` 函数在一些嵌入式系统或操作系统中可以定义为弱符号，以便程序在不使用这些函数时可以链接进库的默认实现。

##### 4.2 **内核模块开发**

在内核开发中，弱符号常用于允许模块扩展或覆盖内核中提供的默认功能。例如，Linux 内核提供了许多弱符号函数，以便内核模块可以根据需要覆盖这些默认的实现。

##### 4.3 **编译器和链接器优化**

编译器和链接器可以使用弱符号来优化代码。例如，编译器在遇到未定义的符号时，可以选择一个弱符号作为默认实现，直到链接时才会确定符号是否需要替换为强符号。

#### 5. **如何定义弱符号**

在 GCC 编译器中，定义弱符号通常通过 `__attribute__((weak))` 来实现。例如：

```c

int __attribute__((weak)) global_var = 10;  // 声明为弱符号
```

或者对于函数：

```c
__attribute__((weak)) void my_function() {
    // 默认实现
}
```

### SystemTap

**SystemTap** 是一个强大的动态跟踪工具，用于在 Linux 系统上动态地插入探针（probes）到内核或用户空间程序中，以便收集性能数据、调试信息、以及监控系统行为。它允许你查看内核和应用程序的内部状态，无需修改源代码或重启系统。SystemTap 可以用于性能分析、错误检测、以及在生产环境中进行系统监控。

#### SystemTap 的主要功能：

1. **内核和用户空间监控**：你可以跟踪系统调用、函数调用、内核事件、文件操作、CPU 状态等。
2. **动态分析**：通过插入探针，在运行时获取系统行为数据，不需要重启机器。
3. **高效性**：SystemTap 对性能的影响很小，尤其是在长期使用时。
4. **灵活性**：你可以编写自己的脚本来分析任何你感兴趣的事件。

#### SystemTap 工作原理：

1. **脚本解析**：编写 SystemTap 脚本，指定你想要跟踪的事件（如函数调用、系统调用等）。
2. **探针插入**：通过编译脚本生成 C 代码，并将其插入到内核或用户空间中，捕获和分析事件。
3. **数据收集**：收集事件数据，可以在脚本中使用打印（`printf`）、计数（`count`）等操作来进行输出和分析。

#### 安装流程

```bash
sudo yum install systemtap systemtap-devel //一定要安装devel包 
sudo yum install kernel-devel kernel-debuginfo //安装内核开发包和debuginfo包 
```

#### 编写一个简单的 SystemTap 脚本

例如，以下是一个简单的 SystemTap 脚本，跟踪系统调用 `open` 并输出打开的文件路径：

### 符号表

符号表（Symbol Table）是计算机程序中用于存储符号信息的数据结构，而 `vmlinux` 并不是符号表本身，但它可以包含符号表。

------

### **1. 符号表是什么**

符号表主要包含程序中使用的各种符号的相关信息，比如变量名、函数名、内存地址、类型信息等。编译器和调试工具需要符号表来解析程序代码，执行链接和调试操作。

#### 符号表的典型内容：

- 符号的名字（如变量、函数、类型等）。
- 符号的类型（变量、函数、类等）。
- 符号所在的位置（内存地址、代码段）。
- 符号的作用域和生命周期（全局、局部）。
- 符号的大小（比如数组大小、变量类型所占字节数）。

#### 符号表的用途：

- **编译器**：在编译阶段解析和检查代码。
- **链接器**：在链接阶段解决符号的地址绑定。
- **调试器**：在调试阶段提供符号名和地址的对应关系，方便开发者调试。

------

### **2. vmlinux 和符号表的关系**

`vmlinux` 是一个包含未压缩内核映像的 ELF 文件（Executable and Linkable Format）。其中可以包含内核的符号表。

#### **符号表在 vmlinux 中的角色**

- **vmlinux 的符号表**记录了内核中所有函数和全局变量的信息。

- 符号表的主要部分包括：

  - `.symtab`：完整符号表，用于调试和链接。
  - `.strtab`：符号名的字符串表。
  - `.dynsym`：动态符号表，仅包含运行时需要的符号。

- 符号表在调试中的作用

  - 帮助调试工具（如 `gdb`、`crash`）解析内核的函数名和变量地址。
- 在内核崩溃分析时，通过符号表可以将地址翻译成人类可读的函数和变量名。

### ARM SPE

ARM SPE（System Performance Events）是一种用于收集和监控系统性能数据的机制，通常用于嵌入式系统和处理器架构中。它允许捕获硬件性能计数器、事件、状态寄存器等多种信息，以便开发者能够监控系统性能、调试瓶颈、优化代码等。SPE可以监测处理器、内存、缓存等系统组件的工作情况，包括但不限于：

- CPU周期数
- 缓存命中与缺失
- 分支预测准确性
- 指令执行情况等

ARM SPE是ARM架构的一部分，主要用于提供硬件性能数据，通常集成在ARM的调试工具中，帮助开发者在嵌入式或低功耗设备中进行优化和故障排除。

#### 为什么ARM SPE存储数据使用环形缓冲区？

ARM SPE机制在存储数据时通常使用环形缓冲区，其原因和优势如下：

##### 1. **高效的数据写入和读取**

环形缓冲区采用循环存储的方式，写入新数据时，数据覆盖最旧的数据。这种方式允许在有限的存储空间内持续接收和存储新数据，而不会因为数据量大而导致溢出或数据丢失。

##### 2. **避免数据丢失**

在高频率的事件数据生成过程中，事件数据可能迅速积累。如果没有适当的存储机制，数据可能会因为缓冲区满而丢失。环形缓冲区能够保证在数据缓冲区已满时，最老的数据会被新数据替代，避免数据丢失。

##### 3. **适应实时需求**

环形缓冲区特别适合实时数据处理，它允许快速地将数据写入并按需读取。ARM SPE常用于实时性能监控，因此环形缓冲区能够高效地支持这些应用，尤其在性能分析和调试时。

##### 4. **资源有限情况下的高效利用**

在嵌入式系统或低功耗设备中，内存资源通常非常有限。环形缓冲区能够在固定大小的缓冲区内进行循环使用，最大限度地提高内存使用效率，不需要动态分配或大量内存管理开销。

##### 5. **简单易实现**

环形缓冲区是一种相对简单且高效的数据结构，能够在资源有限的环境中提供持续的数据流处理。由于其简洁的结构，环形缓冲区可以在硬件或软件层面上实现，且不会引入额外的复杂性。

#### 环形缓冲区的优势与劣势

##### **优势：**

1. **持续数据存储：** 当事件数据量大且频繁时，环形缓冲区确保数据能够持续存储，并且不丢失。
2. **高效读取和写入：** 由于没有频繁的内存移动和分配，读写操作简单且高效。
3. **简单实现：** 环形缓冲区的实现非常简单，尤其适合内存资源有限的嵌入式系统。
4. **低延迟：** 环形缓冲区可以减少数据流的延迟，特别是在需要实时监控或处理的场景下。

##### **劣势：**

1. **有限的存储空间：** 缓冲区的大小是固定的，不能容纳超出其容量的数据。如果事件数据产生速率远高于处理速率，数据会被覆盖或丢失。
2. **不适合存储大数据量：** 环形缓冲区适合存储有限的数据量，若事件数据量非常大，可能需要更多的存储资源。
3. **无法恢复覆盖的数据：** 一旦数据被覆盖，丢失的数据无法恢复，这可能是数据采集中的一个缺点，尤其是在无法承受数据丢失的情况下。

#### 简易的环形缓冲区实现

下面是一个简单的C++实现，用于演示如何实现环形缓冲区：

```cpp
#include <iostream>
#include <vector>

template <typename T>
class CircularBuffer {
public:
    CircularBuffer(size_t size) : buffer(size), head(0), tail(0), full(false) {}

    // 写入数据
    bool write(const T& value) {
        if (full) {
            // 缓冲区已满，覆盖最旧的数据
            tail = (tail + 1) % buffer.size();
        }
        buffer[head] = value;
        head = (head + 1) % buffer.size();
        full = head == tail;
        return true;
    }

    // 读取数据
    bool read(T& value) {
        if (isEmpty()) {
            std::cerr << "Buffer is empty!" << std::endl;
            return false;
        }
        value = buffer[tail];
        tail = (tail + 1) % buffer.size();
        full = false;
        return true;
    }

    // 检查缓冲区是否为空
    bool isEmpty() const {
        return (!full && head == tail);
    }

    // 检查缓冲区是否满
    bool isFull() const {
        return full;
    }

private:
    std::vector<T> buffer;
    size_t head;
    size_t tail;
    bool full;
};

int main() {
    CircularBuffer<int> buffer(5);  // 创建大小为5的环形缓冲区

    // 写入数据
    for (int i = 0; i < 7; ++i) {
        buffer.write(i);
        std::cout << "Written: " << i << std::endl;
    }

    // 读取数据
    int value;
    while (!buffer.isEmpty()) {
        buffer.read(value);
        std::cout << "Read: " << value << std::endl;
    }

    return 0;
}
```

- **`CircularBuffer`类**：该类实现了一个简单的环形缓冲区，具有`write`和`read`方法来写入和读取数据。
- **`write`方法**：将数据写入缓冲区。如果缓冲区已满，写入会覆盖最旧的数据。
- **`read`方法**：从缓冲区读取数据，按照FIFO（先进先出）顺序读取。
- **`isEmpty`和`isFull`方法**：检查缓冲区是否为空或已满。

在上述代码中，环形缓冲区的大小是固定的（5个元素），一旦写入超过缓冲区的容量，最旧的数据会被新数据覆盖。

### 大小端字节序

#### 1. **大端字节序（Big-Endian）**

在大端字节序中，多字节数据类型的最高有效字节（Most Significant Byte，MSB）存储在最低的内存地址上，最低有效字节（Least Significant Byte，LSB）存储在最高的内存地址上。

**举例说明：**

假设有一个 4 字节的整数 `0x12345678`，在大端字节序下，它在内存中的存储顺序是：

```
内存地址     值
0x00       0x12  (最高有效字节)
0x01       0x34
0x02       0x56
0x03       0x78  (最低有效字节)
```

也就是说，高位字节存储在低地址处，低位字节存储在高地址处。

#### 2. **小端字节序（Little-Endian）**

在小端字节序中，多字节数据类型的最低有效字节（LSB）存储在最低的内存地址上，最高有效字节（MSB）存储在最高的内存地址上。

**举例说明：**

假设同样是 4 字节的整数 `0x12345678`，在小端字节序下，它在内存中的存储顺序是：

```
scss复制代码内存地址     值
0x00       0x78  (最低有效字节)
0x01       0x56
0x02       0x34
0x03       0x12  (最高有效字节)
```

低位字节存储在低地址处，高位字节存储在高地址处。

#### 3. **如何判断字节序**

可以通过以下代码在C或C++中判断当前平台的字节序：

```cpp
#include <iostream>

int main() {
    uint32_t num = 0x12345678;
    uint8_t* ptr = reinterpret_cast<uint8_t*>(&num);

    if (ptr[0] == 0x78) {
        std::cout << "Little Endian" << std::endl;
    } else if (ptr[0] == 0x12) {
        std::cout << "Big Endian" << std::endl;
    } else {
        std::cout << "Unknown Endian" << std::endl;
    }

    return 0;
}
```

### tracepoints

### kprobes技术

### Fentry

### uprobes

是 Linux 提供的一种动态追踪机制，用于在用户空间的指定函数或指令上插入探针。它允许开发者实时监控程序执行，进行性能分析或调试，类似于 kprobes，但作用于用户态应用程序。

```c
#include <vmlinux.h>
#include <bpf/bpf_helpers.h>
#include <bpf/bpf_tracing.h>

#define TASK_COMM_LEN 16
#define MAX_LINE_SIZE 80

/* Format of u[ret]probe section definition supporting auto-attach:
 * u[ret]probe/binary:function[+offset]
 *
 * binary can be an absolute/relative path or a filename; the latter is resolved to a
 * full binary path via bpf_program__attach_uprobe_opts.
 *
 * Specifying uprobe+ ensures we carry out strict matching; either "uprobe" must be
 * specified (and auto-attach is not possible) or the above format is specified for
 * auto-attach.
 */
SEC("uretprobe//bin/bash:readline") //宏绑定 
int BPF_KRETPROBE(printret, const void *ret)
{
 char str[MAX_LINE_SIZE];
 char comm[TASK_COMM_LEN];
 u32 pid;

 if (!ret)
  return 0;

 bpf_get_current_comm(&comm, sizeof(comm));//bpf_get_current_comm 函数获取当前任务的名称，并将其存储在 comm 数组中。

 pid = bpf_get_current_pid_tgid() >> 32; //bpf_get_current_pid_tgid 函数获取当前进程的 PID，并将其存储在 pid 变量中
 bpf_probe_read_user_str(str, sizeof(str), ret); //使用 bpf_probe_read_user_str 函数从用户空间读取 readline 函数的返回值，并将其存储在 str 数组中。

 bpf_printk("PID %d (%s) read: %s ", pid, comm, str); //最后使用 bpf_printk 函数输出 PID、任务名称和用户输入的字符串。

 return 0;
};
```

### sigsnoop 

是一个基于 BPF（Berkeley Packet Filter）的工具，用于动态追踪和监控进程中信号的发送和接收。它可以通过 BPF 程序直接在内核中捕获进程发送的信号，以及信号的接收过程，并将这些信息输出到用户空间。这种方式允许你在不修改目标应用程序代码的情况下实时监控信号相关的事件。

```c
#include <vmlinux.h>
#include <bpf/bpf_helpers.h>
#include <bpf/bpf_tracing.h>

#define MAX_ENTRIES 10240
#define TASK_COMM_LEN 16
// 定义一个事件结构体，用来保存进程的相关信息
struct event {
 unsigned int pid;
 unsigned int tpid;
 int sig;
 int ret;
 char comm[TASK_COMM_LEN];
};
// 定义一个 BPF 哈希表，用来保存事件信息
struct {
 __uint(type, BPF_MAP_TYPE_HASH);// 指定为哈希表类型
 __uint(max_entries, MAX_ENTRIES);// 设置最大条目数
 __type(key, __u32);// 键的类型是 32 位无符号整数（线程 ID）
 __type(value, struct event);// 值的类型是 struct event 结构
} values SEC(".maps");  //加载到内核中 可以被内核使用

{

 __u32 tid;

 pid_tgid = bpf_get_current_pid_tgid();
 tid = (__u32)pid_tgid;
 event.pid = pid_tgid >> 32;

 event.sig = sig;
 bpf_get_current_comm(event.comm, sizeof(event.comm));
  //更新value 通过tid 和event     BPF_ANY: 这是一个操作标志，表示插入或更新元素。如果指定为 BPF_ANY，即使键已存在，也会更新现有的值；如果是 BPF_NOEXIST，则只有在键不存在时才插入。
 bpf_map_update_elem(&values, &tid, &event, BPF_ANY);
 return 0;
}

{

 __u32 tid = (__u32)pid_tgid;

cleanup:
     // 删除哈希表中的事件信息

}
```

## 系统调用execve

在Linux程序中，通过调用execve（），进程能够以全新程序来替换当前运行的程序。再次过程中，将丢弃旧有程序，进程的栈.数据以及堆段会被新程序所替换。这个 exec 函数族就提供了一个在进程中启动另一个程序执行的方法。
它根据指定的文件名或目录名找到可执行文件，并用它来代替当前进程的执行映像。也就是说，exec调用并没有生成新进程，一个进程一旦调用 exec函数，它本身就“死亡”了，系统把代码段替换成新程序的代码，放弃原有的数据段和堆栈段，并为新程序分配新的数据段与堆栈段，惟一保留的就是进程的 ID。也就是说，对系统而言，还是同一个进程，不过执行的已经是另外一个程序了。

``` c
static long do_unlinkat(int dfd, const char __user *pathname)
{
    int error;
    struct filename *name; // 存储解析后的路径名信息
    struct dentry *dentry; // 表示目录项（要删除的文件的目录项）
    struct nameidata nd; // 用于路径解析的数据结构
    struct inode *inode = NULL; // 要操作的 inode
    unsigned int lookup_flags = 0; // 路径解析标志

retry:
    // 解析传入路径的父目录，并初始化 nd 和 name
    name = user_path_parent(dfd, pathname, &nd, lookup_flags);
    if (IS_ERR(name)) // 如果路径解析失败
        return PTR_ERR(name); // 返回错误码

    error = -EISDIR; // 默认设置错误为路径类型错误
    if (nd.last_type != LAST_NORM) // 检查路径是否为普通文件（非符号链接或特殊文件）
        goto exit1;

    // 清除 LOOKUP_PARENT 标志，表明解析到具体文件位置
    nd.flags &= ~LOOKUP_PARENT;

    // 检查文件系统的挂载状态，确保写操作是允许的
    error = mnt_want_write(nd.path.mnt);
    if (error) // 如果挂载不允许写操作
        goto exit1;

    // 加锁父目录的 inode，避免并发修改目录内容
    mutex_lock_nested(&nd.path.dentry->d_inode->i_mutex, I_MUTEX_PARENT);

    // 查找并获取目标文件的 dentry
    dentry = lookup_hash(&nd);
    error = PTR_ERR(dentry); // 获取错误值（如果存在）

    if (!IS_ERR(dentry)) { // 如果目标文件的 dentry 查找成功
        /* 为什么在这里检查？因为需要准确的错误码 */
        if (nd.last.name[nd.last.len]) // 检查路径是否多余斜杠
            goto slashes;

        inode = dentry->d_inode; // 获取目标文件的 inode
        if (!inode) // 如果 inode 不存在（可能文件不存在或路径无效）
            goto slashes;

        ihold(inode); // 增加 inode 的引用计数，防止被释放

        // 安全检查：调用安全框架，判断是否允许删除操作
        error = security_path_unlink(&nd.path, dentry);
        if (error) // 如果安全检查失败
            goto exit2;

        // 执行 VFS 层面的 unlink 操作，实际删除文件
        error = vfs_unlink(nd.path.dentry->d_inode, dentry);

exit2:
        dput(dentry); // 释放 dentry 的引用计数
    }

    // 解锁父目录的 inode
    mutex_unlock(&nd.path.dentry->d_inode->i_mutex);

    if (inode) // 如果 inode 不为空
        iput(inode); // 释放 inode 的引用计数（可能触发删除操作）

    ...
exit1:
    // 释放路径资源（例如 mnt 和 dentry）
    return error;
}
```

### **Clang, LLVM 和 LLD 介绍**

#### 1. **Clang**

- **概述**：Clang 是一个基于 LLVM 的 C、C++ 和 Objective-C 编译器前端。它为这三种编程语言提供语法分析、语义分析、优化和代码生成的功能。

- 关键特性

  ：

  - **前端编译器**：Clang 负责将源代码转化为 LLVM IR（中间表示），然后由 LLVM 后端进行优化和代码生成。
  - **错误和警告信息**：Clang 提供非常友好、详细且易于理解的错误、警告和诊断信息，使得开发者能够更快速地定位和解决问题。
  - **兼容性**：Clang 支持 GCC 的编译选项和标志，确保了 GCC 项目的兼容性，同时它也支持 C++11、C++14、C++17 和其他现代 C++ 标准。
  - **性能**：Clang 相对于 GCC 提供更快的编译速度，尤其在增量编译时表现突出。

#### 2. **LLVM**

- **概述**：LLVM（Low-Level Virtual Machine）是一个开放源代码的编译器基础设施项目，提供了用于编译、优化、分析和代码生成的工具链。它并不仅限于 C/C++，也支持多种编程语言，如 Rust 和 Swift。

- 关键特性

  ：

  - **中间表示（IR）**：LLVM 提供一个平台无关的中间表示（LLVM IR），使得编译器可以在不同平台之间共享优化和代码生成的工作。
  - **优化工具**：LLVM 提供了一整套优化工具（如 Dead Code Elimination, Loop Unrolling 等），能够在编译过程中对代码进行深度优化。
  - **跨平台支持**：LLVM 支持多种架构（如 x86, ARM, MIPS 等）和平台（如 Windows, macOS, Linux）。它通过提供不同的后端（Target）来生成特定平台的机器代码。
  - **模块化设计**：LLVM 是一个模块化的系统，允许开发者根据需要扩展或替换各个组件。Clang 就是其中的一部分，负责语言的解析，而其他组件则负责优化和代码生成。

#### 3. **LLD**

- **概述**：LLD 是 LLVM 项目中的链接器，旨在替代 GNU 的传统链接器 ld，提供更快的链接速度和对多种目标文件格式的支持。

- 关键特性

  ：

  - **性能**：LLD 设计的重点之一就是高性能，尤其是在大项目的链接过程中，它显著提高了链接速度。LLD 在处理大型项目时的表现通常比 GNU ld 更快。
  - **兼容性**：LLD 支持多种文件格式，如 ELF、Mach-O、COFF 等，能够与 GCC 和 Clang 配合使用，支持多种操作系统。
  - **与 Clang/LLVM 的集成**：LLD 与 Clang 和 LLVM 紧密集成，使得它们可以共享一些中间表示，进一步提高了链接效率。
  - **支持的功能**：LLD 提供了许多标准的链接功能，如符号解析、符号重定位、生成可执行文件、库的静态链接等。

### **Clang, LLVM 和 LLD 关系**

- **Clang** 是 LLVM 项目的一部分，主要负责源代码的编译和优化，输出中间表示（IR）。
- **LLVM** 是整个编译器基础设施的核心，包含了 Clang 和其他工具（如 LLVM IR 优化、目标代码生成等）。
- **LLD** 是 LLVM 的链接器，替代了传统的 GNU ld 链接器，提供高效的链接过程。

### **优点和特点总结**

- **Clang** 提供了清晰的错误信息和现代化的编译器架构，编译速度通常优于 GCC。
- **LLVM** 提供了强大的跨平台优化功能，并且它的模块化设计使得开发者能够在编译流程中做更多的定制。
- **LLD** 提供了快速的链接过程，并且支持多种目标平台和格式，适用于大规模项目的构建。

### **适用场景**

- **Clang/LLVM** 适用于那些对编译速度、错误报告和现代化工具链有较高要求的开发项目，尤其在 C/C++ 开发中，Clang 作为前端能够提供显著的性能提升。
- **LLD** 适用于需要高效链接的大型项目，尤其是在大型开源项目或者需要频繁构建的开发环境中。

这三者是互相配合的，能够在现代的编译流程中提供强大且高效的支持。

### LTO 编译选项

要启用 LTO，你需要在编译和链接时指定特定的选项。这里是一些常用的 LTO 相关编译选项，主要适用于 Clang 或 GCC 编译器：

1. **`-flto`**

   - 启用 LTO（Link Time Optimization）。这是最常见的选项，适用于编译和链接阶段。

   - 示例：

     ```
     bash复制编辑clang -O2 -flto -c file.c -o file.o  # 编译阶段
     clang -O2 -flto file.o -o output    # 链接阶段
     ```

2. **`-flto=full`**

   - 启用全 LTO，表示将进行完整的链接时优化，可能包括更深入的跨模块优化。

   - 示例：

     ```
     bash复制编辑clang -O2 -flto=full -c file.c -o file.o
     clang -O2 -flto=full file.o -o output
     ```

3. **`-fno-lto`**

   - 禁用 LTO。如果你之前启用了 LTO，这个选项可以关闭它。

   - 示例：

     ```
     bash
     
     
     复制编辑
     clang -O2 -fno-lto -c file.c -o file.o
     ```

4. **`-flto-partition=none`**

   - 禁止对模块进行分区。这个选项可用于控制 LTO 时如何划分模块进行优化。

   - 示例：

     ```
     bash
     
     
     复制编辑
     clang -O2 -flto -flto-partition=none -c file.c -o file.o
     ```

5. **`-fwhole-program-vtables`**

   - 这个选项在开启 LTO 的时候可以用于优化虚拟表的生成方式，特别是对 C++ 中的多态对象的优化。

   - 示例：

     ```
     bash
     
     
     复制编辑
     clang -O2 -flto -fwhole-program-vtables -c file.c -o file.o
     ```

6. **`-LTO-merge`**

   - 使得多个 LTO 模块可以合并。这个选项可以在编译器和链接器之间调整优化行为。

   - 示例：

     ```
     bash
     
     
     复制编辑
     clang -O2 -flto -LTO-merge file.o -o output
     ```

#### LTO 的好处：

- **跨模块优化**：LTO 使得编译器能够跨多个源文件进行优化，通常能带来更好的性能。
- **去除冗余代码**：通过 LTO，编译器能更好地识别和去除不必要的代码，比如内联函数或常量。
- **更好的内存使用**：LTO 可以减少程序的内存使用，特别是当多个模块共享相同代码时。
- **优化跨模块的函数调用**：LTO 能够在链接时消除函数调用开销，进一步优化性能。

### fallthrough关键字 

`fallthrough` 是 C++17 引入的一个关键字，用于在 `switch` 语句中显式指示某个 `case` 标签的代码块会“跳过”并继续执行下一个 `case` 标签的代码。它可以帮助提高代码的可读性，并避免编译器警告。

#### 背景

在 `switch` 语句中，通常一个 `case` 标签下的代码会在执行后跳出 `switch` 语句块，而不继续执行后续 `case` 的代码。但是，有时你可能希望让多个 `case` 标签共享相同的代码，这时可以不加 `break` 语句直接“跳过”到下一个 `case`，这种做法容易导致错误，编译器也可能会给出警告。为了解决这个问题，C++17 引入了 `fallthrough` 关键字。

#### 语法

```
cpp复制编辑switch (expression) {
    case 1:
        // Do something
        [[fallthrough]];  // 显式指示继续执行下一个 case
    case 2:
        // Do something else
        break;
    case 3:
        // Do another thing
        break;
    default:
        // Default action
}
```

#### 关键点

1. **明确意图**：`[[fallthrough]]` 告诉编译器程序员明确地意图是在一个 `case` 中执行完代码后继续执行下一个 `case`，避免了遗漏 `break` 语句时编译器的警告。
2. **避免编译警告**：如果没有 `[[fallthrough]]`，而你在 `case` 中没有 `break`，编译器会发出警告，指示可能发生意外的代码流转。
3. **强制执行**：`fallthrough` 提高了代码的可读性，尤其在多种情况下需要执行相同的操作时。

- **作用**：减少内核启动时的输出信息，仅显示最基本的日志。
- **用途**：适用于普通用户或图形界面启动，让启动过程更加简洁。
- **示例**：`quiet` 会禁止显示大量的内核日志信息。

#### rd.info和rd.debug

这些是内核启动时的一些调试选项。你可以在`GRUB_CMDLINE_LINUX_DEFAULT`中添加它们：

- **rd.info**：用于在启动时输出关于初始化进程的信息。
- **rd.debug**：启用调试输出，用于详细跟踪内核启动过程中的信息。

#### nomodeset

- 禁用内核模式设置（KMS），防止启动时自动设置显示分辨率和图形模式。通常用于图形显示问题或兼容性问题的修复。

### ldd工具 

`ldd` 是 Linux 系统中用于查看共享库依赖的命令。它显示了一个可执行文件或共享库（.so 文件）所依赖的共享库的列表。`ldd` 用于调试和查看二进制程序的依赖关系，帮助开发人员了解程序运行时需要加载的共享库。

### **`-fPIC`（Position Independent Code）**

- 用于 **生成位置无关的共享库（Shared Libraries）**。
- 编译的代码可以在内存中加载到任何地址，而不需要重新定位（Relocation）。
- **推荐用于：共享库（`.so` 文件）**。
- **优势**：在多进程共享时，可以避免同一个库的多份副本，提高内存使用效率。
- **缺点**：对性能有轻微的影响，因为访问全局变量时会使用间接寻址。

### **`-fPIE`（Position Independent Executable）**

- **用于生成位置无关的可执行文件**（而非共享库）。
- 必须配合 `-pie` 链接选项使用，生成一个 **ASLR（Address Space Layout Randomization）** 兼容的可执行文件。
- **推荐用于：需要 ASLR 安全机制的可执行文件**。
- **常见场景**：现代 Linux 发行版默认使用 `-fPIE` 和 `-pie` 选项来增强安全性。
- **效果**：可以防止基于内存地址的攻击（如缓冲区溢出）。

### fork和vfork

### `fork()` 复制的资源

1. **内存地址空间**：
   `fork()` 会为子进程创建一个新的地址空间。父进程和子进程在初期共享相同的物理内存页，但会使用写时复制技术。即，只有在其中一个进程写入该页时，操作系统才会为该进程分配新的内存页。这个机制大大提高了效率，因为如果子进程不修改内存，则父子进程可以共享内存。
2. **文件描述符**：
   父进程的文件描述符表会被复制到子进程。父子进程共享这些文件描述符，意味着它们可以共同访问同一文件或管道。例如，子进程对文件的写操作会影响到父进程，因为它们使用相同的文件描述符。
3. **信号处理**：
   子进程继承父进程的信号处理方式，包括已经安装的信号处理器（例如 `SIGINT` 的处理程序）。但是，父进程的信号屏蔽字（阻塞信号的集合）会被复制到子进程。
4. **环境变量**：
   子进程继承父进程的环境变量。例如，`PATH`、`HOME` 等环境变量会被复制过来。子进程可以在运行时修改这些环境变量。
5. **文件锁和进程间通信（IPC）资源**：
   子进程会继承父进程在文件上设置的文件锁以及某些 IPC 资源（如共享内存、消息队列、信号量等）。
6. **程序计数器和寄存器状态**：
   `fork()` 会复制父进程的程序计数器，堆栈指针等 CPU 状态，但不包括当前执行的代码（这将取决于父进程和子进程的运行环境）。
7. **线程资源**：
   `fork()` 只会复制父进程的主线程（第一个线程），不会复制其他线程。子进程将从父进程的主线程开始执行，并且不会继承父进程的其他线程。

### 不会复制的资源

- **进程ID（PID）**：
  父进程的 PID 和子进程的 PID 是不同的，`fork()` 不会复制进程ID。
- **父进程信息**：
  子进程不继承父进程的 `pid`、`ppid` 等信息，也不会继承父进程的用户 ID、组 ID 等。
- **定时器（如定时任务等）**：
  父进程中的定时器或定时任务（如使用 `alarm` 设置的定时器）不会被复制到子进程中。

### 写时复制（Copy-On-Write, COW）概述

写时复制（COW）是一种优化技术，在 `fork()` 系统调用时，子进程并不会立即复制父进程的内存空间，而是让父子进程共享相同的内存区域。只有在某个进程尝试修改共享内存时，操作系统才会为该进程分配新的内存页面，这样就避免了不必要的内存复制，提高了效率。

### 详细原理

1. **初始共享内存**： 当父进程调用 `fork()` 创建一个子进程时，操作系统会为子进程创建一个新的地址空间，且这个地址空间最初会与父进程的地址空间共享。父子进程的页表（Page Table）会指向相同的物理内存页，这样父子进程可以共享相同的数据。
2. **标记页面为只读**： 父进程和子进程共享的内存页面会被标记为只读。这意味着，父子进程都可以读取这些内存页，但无法修改它们。如果其中一个进程尝试写入这些共享内存页，操作系统会触发一个页面错误（Page Fault）。
3. **发生页面错误时复制页面**： 当进程试图修改共享的内存页时，操作系统会检测到页面错误，并且会为修改的进程分配新的内存页面。这时，操作系统会将父进程的内存内容复制到新的内存页面，并把新页面映射到修改进程的虚拟地址空间中。这样，修改进程和父进程就不再共享同一页面了，只有修改进程会看到新的内容。
4. **内存独立**： 在写时复制的机制下，父进程和子进程的内存内容最终会分离，互不干扰。父进程对内存的修改不会影响子进程，反之亦然。

#### **`vfork()`**

- **功能**： `vfork()` 与 `fork()` 的作用相同，也用于创建子进程。但与 `fork()` 不同的是，`vfork()` 在创建子进程时，**父进程会被挂起**，直到子进程执行完毕并返回。子进程通常会在 `vfork()` 后立即调用 `exec()` 系列函数来加载新的程序，或者调用 `_exit()` 来终止自己。
- **内存行为**： `vfork()` 并不会复制父进程的内存。父进程和子进程会共享同一块内存区域，直到子进程执行完毕。在子进程执行期间，父进程是挂起状态的，不允许对共享内存进行修改。因为子进程的执行依赖于对父进程内存的控制，这样会避免不必要的内存复制，优化性能。
- **性能开销**： `vfork()` 相比 `fork()` 具有更小的性能开销。由于没有进行内存复制，因此它更加高效，尤其是在子进程很快会调用 `exec()` 或 `_exit()` 的情况下。
- **使用场景**： `vfork()` 主要用于那些子进程立即执行 `exec()` 或 `_exit()` 的情况。例如，在实现 `sh` 等 shell 时，子进程通常会在 `vfork()` 后直接加载新程序。由于父进程在子进程执行期间会被挂起，所以 `vfork()` 不能在子进程和父进程共享修改数据的情况下使用。

在 Linux 中，线程创建通常是通过 POSIX 线程 API（如 `pthread_create()`）来实现，而这些 API 的底层实际上调用了 `clone()` 系统调用。`clone()` 允许调用者指定一组标志（flags），用来控制新线程与调用者（父线程）之间共享哪些资源。

**clone() 系统调用**

- `clone()` 是一种灵活的进程创建接口，可以通过指定不同的标志，实现不同程度的资源共享。
- 当用来创建线程时，常用的标志包括：
  - **CLONE_VM**：使子进程与父进程共享同一地址空间（即内存区域）。
  - **CLONE_FS**：共享文件系统信息（当前工作目录、根目录等）。
  - **CLONE_FILES**：共享文件描述符表。
  - **CLONE_SIGHAND**：共享信号处理器。
  - **CLONE_THREAD**：让新进程和父进程属于同一个线程组，从而实现线程概念。

### 进程0和进程1

init进程是被Linux 0进程创建，0进程把init进程fork出来后，就退化成IDLE进程。这个进程，是特殊调度类，所有进程都停止或睡眠后，就会调度进程0运行，此时处于CPU低功耗状态。

### 什么是僵尸进程？

僵尸是子进程死了，资源已经释放。所以不可能有内存泄漏等。但是父进程还没有来得及去wait回收它。task_struct 还在，父进程可以查到子进程的死因。

在 Linux 中，进程的“睡眠”状态通常可以分为两类：

------

### 1. 可中断睡眠（浅睡眠）

- **状态符号**：在 `ps` 命令中通常显示为 **S**（Sleeping）。
- **特点**：
  - 处于可中断睡眠状态的进程正在等待某个事件（例如 I/O、信号或资源变为可用）。
  - 这种状态下，进程可以被信号中断，进而被唤醒。
  - 因为它们能够响应信号，所以通常被称为“浅睡眠”。
- **应用场景**：
  常见于大部分普通等待状态的进程，比如等待用户输入、网络数据或其他系统事件。

------

### 2. 不可中断睡眠（深睡眠）

- **状态符号**：在 `ps` 命令中通常显示为 **D**（Uninterruptible Sleep）。
- **特点**：
  - 进程处于不可中断睡眠状态时，正在等待一些不能被中断的操作完成，比如等待磁盘 I/O 完成。
  - 在这种状态下，即使有信号到达，进程也不会立即响应或被唤醒。
  - 由于等待的资源或操作通常是关键的硬件操作，这种睡眠状态被称为“深睡眠”。
- **应用场景**：
  主要用于需要确保数据一致性的场景，比如在执行磁盘读写、文件系统操作或其它 I/O 操作时，为了防止数据错误，内核会将进程置于不可中断状态，直到 I/O 完成。

### `rq`（可运行队列，Runqueue）

- **功能**：每个 CPU 都有一个 `rq`，用于管理该 CPU 上的可运行进程。它包含了关于 CPU 运行状态的信息。

- 关键成员

  ：

  - **自旋锁**：保护队列数据结构，确保队列操作的原子性，避免竞态条件。
  - **进程数量**：跟踪队列中待调度的进程数量。
  - **CFS 结构体**：`rq` 包含一个 CFS（Completely Fair Scheduler）结构体，用于存储公平调度的信息，支持负载均衡和公平性。
  - **当前运行进程**：指向当前正在该 CPU 上运行的进程描述符。

### **如何实现公平调度**

CFS 通过以下几个机制来实现公平性：

#### (1) **虚拟运行时间（vruntime）**

- **`vruntime`** 是 CFS 调度器的关键指标，它用于衡量每个进程的运行时长。在 Linux 内核中，每个进程都会维护一个 `vruntime` 值。CFS 调度器会根据这个值来选择下一个执行的进程。`vruntime` 较小的进程会被优先调度，进程的 `vruntime` 会随着它的运行时间增加而增加，保证运行时间短的进程有更多的机会被调度。

#### (2) **红黑树**

- CFS 使用红黑树来组织进程，`sched_entity`（调度实体）作为树的节点。每个 `sched_entity` 包含进程的 `vruntime`，并根据该值来排序。通过红黑树，调度器能够高效地找到 `vruntime` 最小的进程，从而选择最公平的进程执行。

#### (3) **时间片与权重**

- CFS 不使用固定的时间片，而是动态调整每个进程的时间片长度。进程的权重（`load_weight`）影响它分配的 CPU 时间。较高权重的进程可以占用更多的 CPU 时间。

#### RCU机制

RCU（**Read-Copy-Update**，读-复制-更新）是一种高效的同步机制，特别适用于读多写少的并发环境。RCU 的目标是减少写操作对读操作的影响，使得读取操作可以在不加锁的情况下并行执行，而写操作通过一定的延迟保证数据一致性。

```c
#include <linux/rcupdate.h>

struct my_data {
    int value;
    struct rcu_head rcu;  // RCPU 保护的数据结构
};

static struct my_data *data;

void update_data(int new_value)
{
    struct my_data *new_data;

    new_data = kmalloc(sizeof(struct my_data), GFP_KERNEL);
    new_data->value = new_value;

    // 通过 call_rcu 延迟回收旧数据
    call_rcu(&data->rcu, free_data);

    // 更新数据指针，指向新的数据副本
    data = new_data;
}

void read_data(void)
{
    struct my_data *local_data;

    // 在读操作期间获取 RCPU 锁
    rcu_read_lock();

    local_data = data;
    pr_info("Data value: %d\n", local_data->value);

    rcu_read_unlock();
}

void free_data(struct rcu_head *head)
{
    kfree(container_of(head, struct my_data, rcpu));
}

```

schedule函数

**关闭内核抢占，标识 CPU 状态**：首先，关闭内核抢占，确保当前进程在进行上下文切换时不会被中断。然后通知 RCU（Read-Copy-Update）更新状态，关闭本地中断，保证调度的原子性。

**检查 `prev` 状态**：检查当前运行的进程 `prev` 的状态，决定是否将其插入到运行队列或从队列中删除。

**调用 `task_on_rq_queued(prev)`**：将当前进程 `prev` 插入到调度队列的队尾（即使它继续保持可运行状态）。

**调用 `pick_next_task`**：选择下一个将要执行的进程 `next`，这是调度算法的核心部分。

**调用 `context_switch(rq, prev, next)`**：进行上下文切换，保存当前进程 `prev` 的状态，并加载即将执行的进程 `next` 的状态。

在 **spec 文件**（通常用于 RPM 包构建）中，`make` 和 `%{make}` 之间有以下本质区别：

1. **`make`**：

   - `make` 是一个固定的命令，直接执行该命令时会调用构建工具 `make` 来构建程序。这个命令不会对环境进行任何额外的处理或扩展，它是硬编码的，表示直接执行 `make` 命令。

   - 用法：通常用于执行构建目标，例如：

     ```
     spec
     
     
     复制编辑
     make
     ```

2. **`%{make}`**：

   - `%{make}` 是一个宏，它在 **RPM 构建系统** 中定义，并自动解析成系统默认的 `make` 命令路径。此宏可能根据不同的操作系统或构建环境有所变化。`%{make}` 会根据当前环境（例如，`CC`、`CFLAGS` 和其他环境变量）来动态解析出适当的 `make` 命令。
   - 这种方式提供了更好的跨平台兼容性，因为它可能会自动选择更适合当前系统配置的 `make` 版本或自定义选项。

### 具体差异：

- **`make`**：直接执行 `make` 命令，不受环境变化影响。
- **`%{make}`**：调用 RPM 构建系统定义的宏，支持环境适配，能根据系统配置调整 `make` 的执行方式（例如路径、编译选项等）。

而该位掩码 , 正是对应 进程task_struct数据结构中的cpus_allowed属性，与cpu的每个逻辑核心一一对应。

### XSI IPC (Inter-Process Communication) System V规范

XSI IPC（X/Open System Interface Inter-Process Communication）是基于 **System V** 标准的一组进程间通信机制，用于不同进程之间共享数据、同步、和互斥操作。它包括了共享内存、信号量和消息队列三种通信手段，属于 **System V IPC** 规范的一部分。

这些通信方式是早期 UNIX 系统中常用的手段，但在 **POSIX** 标准中有些接口进行了改进和更新，使得它们变得更加高效、跨平台和符合现代操作系统的设计理念。尽管如此，System V IPC 仍然在某些场景下有其特殊用途，特别是对旧版应用程序的支持。

### System V IPC和POSIX标准的区别

1. **进程间通信方式**：
   - **System V IPC**：使用了共享内存（Shared Memory）、信号量（Semaphores）和消息队列（Message Queues）这三种手段，操作通常会涉及系统调用，并且需要操作内核来完成。
   - **POSIX标准**：POSIX提供了对多进程和多线程的支持，接口更加现代化，且对于线程间通信一般使用共享内存、信号量等机制，但比System V接口更轻量、灵活。
2. **性能差异**：
   - **System V IPC**：每次对信号量等资源进行操作时，都会发生 **系统调用**，即每次操作都要进入内核，导致较高的开销，特别是在高频操作中，可能会影响性能。
   - **POSIX**：POSIX标准的多线程操作提供了更轻量的机制（如轻量级线程），它尽量避免过多进入内核，优化了线程间的通信与同步效率。
3. **互斥与同步**：
   - **System V信号量**：信号量是一种用于进程间同步的机制，它允许多个进程对共享资源的访问进行协调，防止竞争条件。系统调用 `semop()` 是信号量操作的关键。
   - **POSIX信号量**：POSIX也提供信号量机制，但在实现上可能会对多线程的同步提供更多优化，以便减少内核的干预。

### XSI IPC命令

**`ipcs` 命令** 用于查看当前系统中正在使用的 **System V IPC** 资源，包括共享内存、信号量和消息队列的状态。

- **共享内存**（Shared Memory）：一种用于进程间共享数据的机制，允许多个进程共享一段内存区域。
- **信号量**（Semaphore）：用于控制多个进程对共享资源的访问，避免竞争条件。
- **消息队列**（Message Queue）：用于进程间发送和接收消息。

使用 `ipcs` 命令时，常见的输出包括：

- `shm`: 显示共享内存段的使用情况。
- `sem`: 显示信号量的使用情况。
- `msg`: 显示消息队列的使用情况。
