## 1️⃣ 二叉搜索树（BST）基础

**二叉搜索树（BST）特点：**

- 每个节点最多有两个子节点：左子树和右子树
- 左子树所有节点 < 当前节点
- 右子树所有节点 > 当前节点
- 这样可以快速查找某个 key

**示例：**

```
      10
     /  \
    5    20
   / \   /
  3   7 15
```

- 查找 7：
  - 7 < 10 → 左子树
  - 7 > 5 → 右子树 → 找到 7
- 时间复杂度：O(h)，h 是树高度

## 2️⃣ 平衡二叉树的概念

普通 BST 有个问题：

- 如果插入顺序有序，比如 1,2,3,4,5 → 会退化成 **链表**
- 高度 h = n → 查找复杂度 O(n)

**平衡二叉树**：

- 保证树高度 h = O(log n)
- 常用平衡策略：
  - **AVL 树**：严格平衡，高度差 ≤ 1
  - **红黑树**：弱平衡，高度 ≤ 2*log(n+1)
- STL 的 `set` / `map` 使用 **红黑树**

## 1️⃣ 哈希冲突是什么

- 哈希表通过哈希函数 `hash(key)` 将键映射到桶（bucket）：

  ```
  index = hash(key) % bucket_count
  ```

- **冲突**：不同 key 被映射到同一个桶

  ```
  key1 -> bucket 2
  key2 -> bucket 2   ← 冲突
  ```

- 哈希冲突不可避免，好的哈希函数可以减少冲突概率，但不可能完全消除。

## 2️⃣ 常见解决方式

### (1) 链地址法（Separate Chaining）

- 每个桶维护一个**链表或链式结构**存储冲突元素
- 插入：
  1. 计算桶索引
  2. 插入桶链表末尾或头部
- 查找：
  1. 找到桶
  2. 遍历桶链表找到 key
- **C++ STL 默认实现**：
  - C++11 之前：链表
  - C++17 之后：当链表过长时，自动树化（红黑树）提高效率
- 优点：简单，不影响其他桶
- 缺点：某些桶长链表时查找变慢 O(n/bucket_count)

**示意：**

```
桶数组: [0] [1] [2] [3]
桶2链表: key1 -> key2 -> key5
```

------

### (2) 开放寻址法（Open Addressing）

- 所有元素存储在桶数组中，不使用链表
- 冲突时在数组中寻找下一个空桶（探查 sequence）
- 常见策略：
  1. 线性探查（linear probing）
     - 下一个空桶 = (hash + 1) % N
  2. 二次探查（quadratic probing）
     - 下一个空桶 = (hash + i²) % N
  3. 双重散列（double hashing）
     - 下一个桶 = (hash1 + i*hash2) % N
- 优点：内存连续，cache-friendly
- 缺点：删除复杂，需要特殊标记空桶，负载高时性能下降

> C++ STL 并没有使用开放寻址，主要用链地址法。

## 3️⃣ 哈希表性能优化

- **减少冲突**：

  - 选择良好的哈希函数（`std::hash`）
  - 合理设置桶数（`reserve(n)` 或 `rehash(n)`)

- **桶树化**（C++17）：

  - 如果单个桶链表长度超过阈值（如 8），链表变红黑树
  - 查找复杂度从 O(k) → O(log k)

- **负载因子（load factor）**：

  ```
  load_factor = num_elements / bucket_count
  ```

  - 当 load_factor > 最大值（默认 1.0） → 自动扩容 rehash
