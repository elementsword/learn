## 同步

# 🌟 1. atomic —— 最轻量、最快的同步方式（无锁）

### 使用场景

只保护 **一个整数 / 一个指针 / 一个简单 flag 的读写**。

### 为什么快？

atomic 的 ++ 直接翻译成单条机器指令：

x86 下：

```
lock xadd [i], 1
```

ARM 下：

```
LDAXR / STLXR 循环
```

完全不需要 OS 的参与。

### 示例

```
atomic<int> counter = 0;

void work() {
    counter++;
}
```

### 适合：

- 计数器
- 引用计数（shared_ptr 底层）
- 轻量级标志位（stop = true 的那种）
- lock-free 算法

# 🌟 2. mutex —— 最通用的同步方式（互斥锁）

### 使用场景

保护 **复杂的临界区**：

- 修改多个变量
- 操作容器（vector/map）
- 涉及 IO、逻辑

### 示例

```c++
#include <iostream>
#include <thread>
#include <mutex>
using namespace std;

int counter = 0;
mutex m;

void worker() {
    for (int i = 0; i < 1000000; i++) {
        lock_guard<mutex> lock(m);
        counter++; // 受保护
    }
}

int main() {
    thread t1(worker);
    thread t2(worker);
    t1.join();
    t2.join();

    cout << counter << endl;  // 必然是 2000000
}

```

### 性能定位

- 比 atomic 慢（可能进入内核）
- 但可保护任意复杂逻辑
- 最常用同步工具

### 底层依赖：

Linux 下 → **futex**

第一次竞争可能自旋、再不行会调用 futex 进入内核休眠。

# 🌟 3. condition_variable —— 条件变量（等待/唤醒，实现线程通信）

### 使用场景

一个线程等待条件，另一个线程通知。

最典型：生产者–消费者队列。

### 示例

A线程

```
mutex m;
condition_variable cv;
bool ready = false;

void worker() {
    unique_lock<mutex> lock(m);
    cv.wait(lock, [] { return ready; });
}
```

B线程

```
{
    lock_guard<mutex> lock(m);
    ready = true;
}
cv.notify_one();     // 或 notify_all()
```



### 特点

- 带自动解锁/加锁
- 必须配合 mutex 使用
- wait 进入内核，适合长时间等待



# 🧠 wait 内部到底做了什么？

这是你最关键想懂的部分。

### `cv.wait(lock)` 的真实流程：

1. **自动 unlock(mutex)**
    因为你要睡眠，不能占着锁。
2. **调用 Linux futex() → 将线程挂起（阻塞）**
    线程进入睡眠状态，不占 CPU。
3. 另一线程调用 `notify_one()` 时：
   - 修改了共享状态（例如 `ready = true`）
   - 调用 futex_wake 唤醒睡眠线程
4. 被唤醒后：
    **自动重新 lock(mutex)**
    保证醒来的那一刻仍有互斥保护。

# 🌟 4. shared_mutex（读写锁）

### 多读单写

适合大量读、少量写的场景，例如：

- 配置表
- 大字典
- 缓存

### 示例

```
shared_mutex sm;

void reader() {
    shared_lock<shared_mutex> lock(sm);
}

void writer() {
    unique_lock<shared_mutex> lock(sm);
}
```

### 性能

- 多读并发、写独占
- 比 mutex 更复杂，但很好用

### 🔹 内部机制本质

`shared_mutex` 内部一般会维护两个核心状态：

1. **读计数器（reader count）**
   - 当前有多少线程持有共享锁
2. **写标志（writer flag）**
   - 是否有线程持有独占锁

操作逻辑大致是：

```
lock_shared() {          // 读锁
    if (!writer) {
        reader_count++;
    } else {
        等待 writer 释放
    }
}

unlock_shared() {
    reader_count--;
    如果 reader_count == 0 并且有 writer 等待，唤醒 writer
}

lock() {                 // 写锁
    等待 reader_count == 0 并且 writer == false
    writer = true
}

unlock() {
    writer = false
    唤醒等待线程
}
```

也就是说，本质就是 **一个带计数器的互斥锁**，
 用来管理**读共享/写独占**状态。

# 🌟 5. spinlock（自旋锁）

### 特点

线程不睡眠，而是 **忙等**。

适合：

- 临界区非常短（几十纳秒）
- 多核 CPU
- 内核代码、锁竞争很低

### 示例

C++ 没内置，要自己写：

```
std::atomic_flag lock = ATOMIC_FLAG_INIT;

void lock() {
    while (lock.test_and_set(std::memory_order_acquire))
        ; // spin
}

void unlock() {
    lock.clear(std::memory_order_release);
}
```

### 缺点

- 如果临界区长 → CPU 白白浪费
- 用户态一般不用

# 🌟 6. semaphore（信号量）

C++20 加入 `std::counting_semaphore`

### 作用

限制资源数量。

例如：

- 限制线程池最大线程数
- 限制同时下载的连接数

### 示例：

```
counting_semaphore<5> sem(5);

void worker() {
    sem.acquire();
    // do work
    sem.release();
}
```

比 mutex 多了“计数”功能。

# 🌟 7. barrier（屏障）

所有线程必须到达 barrier 才能继续。

### 常见于：

- 并行计算
- MPI（你应该很熟）
- 并行矩阵运算

C++20：

```
std::barrier sync_point(4);

void f() {
    // do something
    sync_point.arrive_and_wait();
}
```

------

# 🌟 8. latch（一次性的屏障）

功能类似 barrier，但只能使用一次。

常用于：

- 等待所有任务启动完毕
- 等待某些初始化完成

```
latch done(10);

void f() {
    // ...
    done.count_down();
}

int main() {
    done.wait();
}
```

# 🌟 9. futex（Linux 内核级原语）*（了解即可）*

mutex/condition_variable 最底层都依赖：

```
futex(addr, FUTEX_WAIT, ...)
futex(addr, FUTEX_WAKE, ...)
```

futex 优化：

- 无竞争时走用户态（快）
- 有竞争才进入内核（慢）

所以 mutex 并不总是慢。
